    JAVA源于随机数生成的规范建议

    生成随机数是一个常用的功能, JAVA/JVM也提供很多不同的选择让开发者决定如何生成随机数.

    但是不同的选择会产生不同的后果, 最后影响的不仅仅是开发, 测试,流程,运维,安全都会产生连锁反应.因此需要在此讨论,澄清,并提供规范的建议. 以下是开发实现和安全运维的建议.

    1. 生成规范的考虑原则

    JAVA有以下几个生成随机数的方式:

    • Random
    • TheadLocalRandom
    • SecureRandom

    关于规范应从以下几点考虑:

    • 随机数可预测性
    • 性能
    • 运维交付复杂度
    • 质量控制

    2. 可预测性

    Random和ThreadLocalRandom都是可预测的, 两者使用一致的弱伪随机算法. 使得攻击者可以模拟出下一个随机数的值. 在不指定种子的情况下,使用系统时间作为种子.

    SecureRandom是更安全的生成随机数的方式. 在不同的操作系统下使用默认SecureRandom()会使用不同的算法.

    SecureRandom random = new SecureRandom();
byte[] values = new byte[20];
random.nextBytes(values);

    比如windows下使用SHA1PRNG, 而linux下使用NativePRNG. 也可以通过SecureRandom.getInstance方法指定算法. 

    SecureRandom secureRandom3 = SecureRandom.getInstance("SHA1PRNG");
SecureRandom secureRandom2 = SecureRandom.getInstance("SHA1PRNG", "SUN");

    完整的操作系统支持随机数生成算法列表在 这里. 其中在linux和macos下, 支持以下几种:

    1. NativePRNG** Sun
    2. SHA1PRNG** Sun
    3. NativePRNGBlocking Sun
    4. NativePRNGNonBlocking

    SHA1PRNG是纯java实现, 在弱种前提下,仍然是可以被跟踪的. 如果对SHA1PRNG的随机性有疑问, 需要使用Native的PRNG算法.

    NativePRNG的相关算法都是从操作系统获得随机种子, 在linux和macos下/dev/random和/dev/urandom可以获得的随机种子. 算法和种子文件的关系如下:

    • NativePRNG (nextBytes() uses /dev/urandom, generateSeed() uses /dev/random)
    • NativePRNGBlocking (nextBytes() and generateSeed() use /dev/random)
    • NativePRNGNonBlocking (nextBytes() and generateSeed() use /dev/urandom)

    /dev/random和/dev/urandom的区别下面提到

    虽然man里面提到random的随机性大于urandom,但是有文章指出,两者没有区别.

    所以以下安全建议是需要参考:

    1. Random和ThreadLocalRandom都是可以简单跟踪的, 随机性和安全性很低
    2. SHA1PRNG是windows下默认的SecureRandom算法, 随机性稍高.
    3. /dev/random和/dev/urandom随机性是一样的. 建议使用基于/dev/urandom的算法, 虽然官方更认同/dev/random
    4. 使用SecureRandom, 建议使用getInstance或getInstanceStrong,指定算法,而不要使用默认的构造函数.因为开发机是windows, 测试和生产机是linux

    3. 性能

    所有随机api都是继承Random, Random是线程安全的, 所以大家都是线程安全的.

    但是多线程下,Random的性能不好,JDK 1.7以后引入了ThreadLocalRandom. 所以多线程环境下, 不能使用Random, 而要用ThreadLocalRandom.

    使用/dev/random的场景下, 阻塞是比较严重的,可以通过在shell下执行以下命令体验一下到底有多严重:

    head -n 1 /dev/random

    /dev/urandom是非阻塞的, 即使表面上随机性不如/dev/random, 但是实际情况可能并不是这样.

    所以以下性能建议是需要参考的:

    1. 尽量不要使用Random,即使在随机性要求不高的情况下
    2. 使用SecureRandom, 建议使用getInstance或getInstanceStrong,指定算法,而不要使用默认的构造函数.因为开发机是windows, 测试和生产机是linux
    3. 如果不能忍受阻塞, 不要使用基于/dev/random的算法获取随机数
    4. 高质量的随机性值得花费CPU资源去获取, 即使在多线程的环境下, 同一个场景使用同一个全局SecureRandom是可以被接受的,即使阻塞也只出现在对象实例化阶段.
    5. 在一些特定场景下, 通过使用命令行参数或者修改JRE里面java.security文件,可以重定向默认构造函数的算法以及将/dev/random的算法重定向到/dev/urandom里 .

    命令行参数覆盖举例:

    -Djava.security.debug="provider,engine=SecureRandom"

    修改java.security举例: 

    securerandom.source=file:/dev/urandom
securerandom.strongAlgorithms=NativePRNGBlocking:SUN

    3.运维交付

    运维交付建议:

    1. 对于现有第三方系统, 在出现阻塞的情况下, 通过命令行测试验证是securerandom阻塞问题后,建议修改java.security将source重写到/dev/urandom上解决问题.
    2. 对于自有系统,建议将算法和source直接修改成非阻塞的方式,制作进模板机.

    4. 质量控制

    质量交付建议:

    1. sonar保留对所有系统Random漏洞的最高级别报警
    2. 对于所有系统应保留对ThreadLocalRandom的漏洞报警,级别可以降低
    3. 对于登陆类,促销活动, 金融类, 券凭证等安全级别较高系统,应对非SecureRandom类以及SHA1PRNG算法提示漏洞
